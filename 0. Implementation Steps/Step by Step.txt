                *********** Unique Contraints IMPLEMENTATION ***********
Implement Unique Constraint: taking eg of userRoles a single user and role can appear multiple times and the db will just accept it
                but a User cannot have the same role twice so we use UniqueContrains such that implementing it make the implemented feilds Unique which in our case is user and role pair
        @Table({tableName: 'user_role',
                indexes: [
                {
                unique: true,
                fields: ["userId", "roleId"]
                }]})
                *********** INDEXING IMPLEMENTATION ************
Implement INDEXING: db searches each row at a time to check conditions and return the matched data
                but this is really slow for large DB's with hundreds of thousands of rows
An INDEX is like a sorted lookup structure: Think of like a book index page: X -> page10, Y -> page 20, Z -> page 50
                Hence we dont need the entire book(DB) we just jump to specific page(INDEX)
Structure: id index eg: 1 -> row pointer 2 -> row pointer i.e give each id an index 
Real DB use structure called B-TREE (It was a Binary Search Tree in GPT example)
Imagine Millions of Rows in UserRole and we need to find where userId is 25..Instead of searching one by one jump to where userId is 25 for users..

Implementation: making userId and roleId indexed:
        @Table({
                tableName: "role_permission",
                indexes: [
                {
                unique: true,
                fields: ["roleId", "permissionId"]//composite key already acts as an index but we still implment index down for faster queries like where userId = ?
                },
                {
                name: "provide_idx_name_or_sequelize_gives_random_name"
                fields: ["roleId"] //indexing roleId
                },
                {
                name: "idx_role_permission_role"
                fields: ["permissionId"]
        }]})

                *********** SEEDERS IMPLEMENTATION ************
Seeding is basically populating the Database with default values on startup
Problems with Manually Adding Roles/Permissions in the Database: 
        Problem 1 — Environments Become Inconsistent: In professional app we have many environments(eg. production stagin local)If manually inserts app behaves differently for diff environments
        Problem 2 — Deployments Break: Imagine adding a new permission in code but it doesnt exist in db
        Problem 3 — New Developers Cannot Run the App: someone clones your project. the roles will be missing
        Problem 4 — Hard to Track Changes: we later change seller can also order products but with manual db change we cant keep track of the change on git
Professional Solution: DATABASE SEEDING
        A SEED script automatically inserts default data into DB.These run automatically during setup
What to Seed: Any kind of static data that is commonly shared between entities or is mainly used for testing
